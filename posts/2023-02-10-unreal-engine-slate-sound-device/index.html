Adding a Custom Slate Sound Device in Unreal Engine

Engine Version: 4.27.2

Prerequisites

  1. Experience with writing C++ for Unreal Engine
  2. Unreal Engine compiled from source
  3. Familiarity with Slate and UMG
  4. Editor module for project

Why Replace the Default Slate Sound Device?

There are a couple reasons to implement a custom ISlateSoundDevice. Perhaps some extra logic around playing a sound in Slate is required. Or maybe a third party audio library being integrated, such as Wwise, uses custom sound classes that don’t adhere nicely to the API defined by USoundBase. The latter is the situation I ran into.

Unreal’s included UMG widgets, such as buttons and checkboxes, have some useful fields related to audio. For example, FButtonStyle includes PressedSlateSound and HoveredSlateSound (“Pressed Sound” and “Hovered Sound” in Blueprint). These fields are of the type FSlateSound:

/**
 * An intermediary to make UBaseSound available for Slate to play sounds
 */
USTRUCT(BlueprintType)
struct SLATECORE_API FSlateSound
{
// <lines removed>

protected:
	/**
	* Pointer to the USoundBase. Holding onto it as a UObject because USoundBase is not available in Slate core.
	* Edited via FSlateSoundStructCustomization to ensure you can only set USoundBase assets on it.
	*/
	UPROPERTY(EditAnywhere, BlueprintReadWrite, Category=Sound, meta=(DisplayName="Sound", AllowedClasses="SoundBase"))
	UObject* ResourceObject;

// <lines removed>
}

As the comment above ResourceObject notes, the type of UObject pointed at is expected to be USoundBase. An FSlateSound is played like this:

FSlateApplication::Get().PlaySound( PressedSound );

FSlateApplication::PlaySound then uses the SlateSoundDevice:

void FSlateApplication::PlaySound( const FSlateSound& SoundToPlay, int32 UserIndex ) const
{
	SlateSoundDevice->PlaySound(SoundToPlay, UserIndex);
}

SlateSoundDevice is an object which implements the interface ISlateSoundDevice. The default Unreal implementation of ISlateSoundDevice is called FSlateSoundDevice. I recommend taking a look at the implementation of FSlateSoundDevice::PlaySound to understand how it plays the ResourceObject contained in FSlateSound.

Keeping all this in mind, the options for playing a custom sound class from a built-in UMG widget are:

  1. Make the custom sound class inherit from USoundBase and implement USoundBase’s API correctly. This is pretty clean if possible, but in some cases the USoundBase API can’t be fit to requirements of the custom sound class.
  2. Listen for the appropriate event in Blueprint/C++ (e.g. OnClicked for buttons) and play the sound in response. This is unwieldy, as it requires duplicating a call to play the custom sound class for each widget instance.
  3. Edit the UMG C++ classes (e.g. UButton) so that instead of FSlateSound they use the custom sound class. This requires editing multiple engine classes in a pretty involved way, plus it creates an engine level dependency on the audio library containing the custom sound class.
  4. Make subclasses of UMG widget classes. This can be done, although inheriting from UMG widgets can be tricky. It also leaves the old sound variables like PressedSlateSound present, so someone working on UI Blueprints might be confused as to which field they should use.
  5. Make FSlateSound point at the custom sound class and create an implementation of ISlateSoundDevice which plays the custom sound class. This is the approach I went with.

FSlateSound Modification

This is the only change to engine code. In FSlateSound, add the name of the custom sound class to AllowedClasses above ResourceObject, like this:

UPROPERTY(EditAnywhere, BlueprintReadWrite, Category=Sound, meta=(DisplayName="Sound", AllowedClasses="SoundBase,CustomSoundClass"))
UObject* ResourceObject;

What is AllowedClasses? As described here, this specifier determines the type of assets displayed in the editor’s asset picker for this particular field. In order to select assets of the custom sound class when editing widgets this specifier has to include the custom sound class. Luckily, AllowedClasses does not create a compile time dependency, so the name of the custom sound class can be inserted even though this module (SlateCore) does not have a dependency on the custom audio library. Note that AllowedClasses expects the “display” name of the class, which is not prefixed. So it uses “SoundBase” instead of USoundBase, for example. Once this change is made the editor can be compiled and opened, a button widget can be made, and its “Pressed Sound” can be set to the custom sound class! But if tested, the sound won’t play yet…

Implement Slate Sound Device

Now an ISlateSoundDevice implementation that knows how to play the custom sound class is needed. First, create an entirely new C++ class which inherits from ISlateSoundDevice:

#pragma once

#include "Sound/ISlateSoundDevice.h"

struct FSlateSound;

class FCustomSlateSoundDevice : public ISlateSoundDevice
{
public:
	virtual ~FCustomSlateSoundDevice() override {}

	virtual void PlaySound(const FSlateSound& Sound, int32 UserIndex = 0) const override;
	virtual float GetSoundDuration(const FSlateSound& Sound) const override;
};

(Unfortunately it is not possible to inherit from FSlateSoundDevice without making an engine change, as FSlateSoundDevice is not publicly exported by its module.)

Note that ISlateSoundDevice is not an Unreal interface, even though it is prefixed with “I”. It is just a C++ abstract class. Because PlaySound and GetSoundDuration are both declared as pure virtual functions in ISlateSoundDevice, they will need to be implemented in FCustomSlateSoundDevice:

void FCustomSlateSoundDevice::PlaySound(const FSlateSound& Sound, int32 UserIndex) const
{
	UObject* const Object = Sound.GetResourceObject();
	if (!Object)
	{
		return;
	}

	USoundBase* const SoundResource = Cast<USoundBase>(Object);
	if (SoundResource)
	{
		// Copied from FSlateSoundDevice::PlaySound
		if (GEngine)
		{
			FAudioDeviceHandle AudioDevice = GEngine->GetActiveAudioDevice();
			if (AudioDevice)
			{
				FActiveSound NewActiveSound;
				NewActiveSound.SetSound(SoundResource);
				NewActiveSound.bIsUISound = true;
				NewActiveSound.UserIndex = UserIndex;
				NewActiveSound.Priority = SoundResource->Priority;

				AudioDevice->AddNewActiveSound(NewActiveSound);
			}
		}

		return;
	}

	UCustomSoundClass* const CustomSound = Cast<UCustomSoundClass>(Object);
	if (CustomSound)
	{
		// <Code to play sound>

		return;
	}

	// An Object but no SoundResource/CustomSoundClass means that the FSlateSound is holding an invalid object; report that as an error
	UE_LOG(LogSlateSoundDevice, Error, TEXT("A sound contains a non-sound resource '%s'"), *Object->GetName());
}

float FCustomSlateSoundDevice::GetSoundDuration(const FSlateSound& Sound) const
{
	UObject* const Object = Sound.GetResourceObject();
	if (!Object)
	{
		return 0.0f;
	}

	USoundBase* const SoundResource = Cast<USoundBase>(Object);
	if (SoundResource)
	{
		return SoundResource->Duration;
	}

	UCustomSoundClass* const CustomSound = Cast<UCustomSoundClass>(Object);
	if (CustomSound)
	{
		return /* <Code to get duration> */;
	}

	return 0.0f;
}

There are a couple details of PlaySound worth noting:

  • There is no access to UWorld in PlaySound. This is expected, as Slate does not depend on a valid World existing. In fact, as the editor itself is Slate, the editor uses ISlateSoundDevice to play audio. So once this change is made, the editor will play sounds through the custom implementation of ISlateSoundDevice. If playing editor sounds is required, make sure to support USoundBase!
  • PlaySound is only ever called on the “game” thread.
  • The UserIndex argument appears to be unused. As far as I can tell no call to PlaySound ever passes in a UserIndex, so it will always be zero, which is the default value. In addition, FSlateSoundDevice passes UserIndex along to FActiveSound, but it’s never used in a meaningful way in the audio code. If I had to guess, I think this is a legacy argument which was intended to differentiate users on a single instance of Unreal (splitscreen) but it is now unused (or was never used). For now, it can be ignored.

Initializing Slate Sound Device

The engine initializes FSlateSoundDevice in UEngine::Init in the following block:

FSlateApplication& CurrentSlateApp = FSlateApplication::Get();
CurrentSlateApp.InitializeSound( TSharedRef<FSlateSoundDevice>( new FSlateSoundDevice() ) );

As covered, ISlateSoundDevice is not an Unreal interface, and FSlateSoundDevice is not a UObject. In fact, this system does not use any of the nice TSubclassOf configuration that other systems use. Instead, something a bit more manual is required.

First, create subclasses of UGameEngine and UUnrealEdEngine as described here. Then override Init in both and after the call to Super::Init put the following:

if (!IsRunningDedicatedServer() && !IsRunningCommandlet())
{
	FSlateApplication& CurrentSlateApp = FSlateApplication::Get();
	CurrentSlateApp.InitializeSound( TSharedRef<FCustomSlateSoundDevice>( new FCustomSlateSoundDevice() ) );
}

This will overwrite the existing FSlateSoundDevice with the custom sound device class. Since FSlateSoundDevice is stored with a shared ref, once FSlateApplication is not referencing the old FSlateSoundDevice it will be garbage collected. This does result in some wasted work creating and then destroying FSlateSoundDevice, but FSlateSoundDevice does not do much in its constructor or destructor so it shouldn’t be too bad.

Conclusion

We have successfully implemented ISlateSoundDevice, and can play custom sounds! Load into the editor, create a button, set “Pressed Sound” to the custom sound asset, and test!
