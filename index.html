Ray Tracer in Rust

GitHub Link

This is a moderately complex ray tracer with a fair number of features. This project fulfilled two primary personal goals. One was to familiarize myself with Rust by building a complete program that touches on a number of different language features (architecture, performance, multithreading, etc.). The other was to construct a more mature ray tracer than my previous attempts. While there are many more features, optimizations, and refactors I would like to implement at some point, I am satisfied with the current code for “V1” of this project.

Features
  • Basic shapes
    • Sphere
    • Triangle
  • Basic materials library
    • Lambert
    • Metal
    • Dielectric
  • Custom JSON scene specification format
    • Supports reading from .obj meshes
  • Bounding volume hierarchy accelerates collision detection
    • Uses Surface Area Heuristic (SAH)
  • Diffuse lights
  • Textures
    • Solid
    • Perlin Noise
    • Images
  • Transformations allow scene manipulation
  • Basic volumes
  • Bump mapping
  • Monte Carlo importance sampling
  • Multithreaded
    • Number of threads is an optional command line argument
  • Basic stats and progress report
Wishlist

    • Point

  • Different texture mappings
    • Spherical
    • Cylindrical
  • True BSDF support
  • Read .mtl files

Mazu

Role: Lead Programmer
Awards:
  • Indie Category Winners - MassDiGI Game Challenge 2020
  • Finalist - MassDiGI Game Challenge 2019

Mazu is a narrative driven genre mash-up, featuring bullet hell and hack n' slash inspired fast-paced gameplay. The title is still under active development by a subset of the original team, and is eyeing a 2020 full release. The current work is centered around content generation, polish, iteration, and bug fixing. Unfortunately, I am no longer actively working on the title, and have not been since June of 2019. As the lead engineer on the project for nearly a year prior to that, however, I established many of the original systems that the game is built on.

The list of interesting parts I wrote includes:

  • Fully procedural “dungeon” generation
    • Constructed rooms and routes that felt natural but purposeful, with branching pathways and hidden rooms
    • Appropriate enemy placement ensured difficulty progressed smoothly and gameplay flow was preserved
    • Seamlessly merged in handmade content when configured to provide story progression
  • Fully configurable cutscene and events system
    • Allowed story to be told in-engine, smoothly integrated with gameplay
    • Enabled fast prototyping of in-game pre-scripted moments
  • Configurable quest system
    • Designers could quickly write new quests with varied objective types
    • In-game quest tracking would automatically provide player objectives based on current quest state
    • Quest system integrated with events system to provide flexible tools for content generation
  • State based player action system, which allowed designers to tune inputs to execute behavior exactly as desired, important in a fast-paced action title
  • Modular and configurable AI behavior trees, providing fast construction of diverse enemy characters
  • Efficient “save anywhere” system, which supported fully dynamic save points while serializing only a small amount of data

For more information about the game, please visit https://www.mazugame.com.

Flow Field Pathfinding

The idea with this project was to implement a version of flow field pathfinding as described in this paper. Java with Processing was possibly not the best choice for performance, but it offers a simple library for visual output and I wanted to prototype quickly without worrying about those details. I believe this flow field implementation is pretty successful, and the "extra features" such as formations work well, but individual unit steering does leave something to be desired. That, however, was not the focus of the project, and is really here to aid in providing a visual demonstration of the algorithm.

Some Quick Features:
  • Flow field pathfinding which subdivides the map into "sectors" of a configurable size
    • A per sector flow field cache that is shared amongst all units, which means units navigating in a similar manner through a given sector can use the same calculated flow field
    • A* search to chain sectors for unit navigation
  • Navigation optimizes for differences in terrain "height", or difficulty of moving through that terrain
    • Also supports navigating around impassable obstacles
  • A formation system that allows grouping arbitrary numbers of units together with a specified separation pattern
    • The formation system is very modular, and adding new formations requires a single new function
    • Formations are dynamic and will reform correctly if units are added or removed
  • Units have a very simple collision avoidance behavior that sends out a single raycast and applies an avoidance force if the unit will collide with something

Go Particle System

This is a fun project I did as part of learning the Go programming language. It's similar to others I have done in Processing and C++, but this is my favorite version. It's fairly basic, allowing users to place new particles into the system with a left click, and to add some force with a right click. It also allows users to make "static" particles (ones that absorb force) by holding the 'S' key while clicking. And there's a secret input that converts all the particles to donuts. I'll let you figure out which key that is...

It uses SDL2 Go bindings by veandco for the graphics, windowing, and inputs.

Space Pirate Loot Shooter

The game places the player as a space pirate attacking convoys of enemies. The player collects loot from the destroyed enemies and brings it to stations to sell. The game was built in Unity. All code was written by me in C#. The models were taken from Turbosquid.

The game features multiple different enemy AI types, including:

  • An enemy who will attempt to ram the player
  • An enemy who will shoot at the player from a certain distance
  • A convoy enemy who will run away from the player

The AI also has a patrol mode, where they are unaware of the player and will go about their own business, and a combat mode, where they will follow whatever their specified combat actions should be.

The other major feature of the game is a fully fleshed out inventory system. The player can collect weapons, armor, and shields, and equip and destroy these components as needed. Different equipment also has different effects. Weapons may offer varying functionality or faster fire rates, for example. Shields defend the player and recharge over time, whereas armor is degraded permanently and the actual item will be destroyed if the player takes too much damage. The player can also visit space stations to buy and sell items, which features a separate menu system.

For controls, use WASD for movement and the mouse to aim and shoot. Press “i” to bring up the inventory screen, “escape” to bring up the pause menu, and “space” when near a station to interact with the space station. In menus, click on the items to swap them or equip them or sell them.
